# 提案機会検知くん リアルタイム版（v2.2・議事録単位 distinct 生成＋領域カテゴリ・グルーピング）

amptalk のコール文字起こし解析（要約minute＋ネイティブ提案機会検知）から、
**提案機会・競争リスク・タスク** を検知し、通知チャンネルに **議事録単位** で送信する。

## チャンネルID（固定）
- `通知先`: `C0AHUC1VDDK` (`#dxm_提案機会_検知くん`)

## 目的（思想）
- **MTG の中でアップセル機会・提案機会を拾って通知する**
- 提案すれば取れるはずの案件を取りこぼさない
- マネ＋AMが「ここ提案できる」「ここ守らないと」「これやる」を把握できる体制をつくる
- 予算内/外の最終判別はマネ＋AMの人の目に委ねる。AIは**機会を広めに拾って渡す装置**に徹する

---

## 処理フロー（Pythonが担当する部分）

1. **コール取得**: amptalk API からコールを取得し、`title` プレフィクスで社外だけ抽出
2. **パース**: amptalk の「要約minute」（Slack議事録Bot投稿と同形式）を ▼決定事項 / ▼タスク<ナイル> / ▼BANT / ▼メモ にセクション分解。**▼タスク<顧客>は検知対象から除外**
3. **重複排除（議事録単位）**: 既処理 call_id をスプシL列から引いて、新規コールだけ処理
4. **AI判定（このスキルが呼ばれる箇所）**: **議事録を丸ごと渡し、重複を排した distinct な項目リストを生成**
5. **通知＆スプシ書き込み**: **💰 か 🚨 が0件の議事録（● だけ）は親投稿もスプシも完全スルー**

---

## Phase 5: AI判定（あなたの役割）＝議事録まるごと → distinct な検知項目リスト

あなたは **1つの商談（議事録）の全内容** を受け取る。そこから、
**重複を排した distinct な 機会(💰)/リスク(🚨)/タスク(●) のリスト** を生成する。

### ★最重要ルール：重複を1件に統合する（過分割を出さない）

amptalk の要約minute は、**同じ論点を ▼決定事項・▼タスク・▼メモ に繰り返し書く**癖がある。
同じ機会/リスク/タスクが複数箇所に出てきても、**必ず1件に統合**すること。

- ❌ 同じアップセル機会を逐語コピーで2〜3件出す（＝過分割。絶対禁止）
- ❌ 「SEO/LLMで300万の追加提案」と「LLM対策の提案余地」のような**言い換え重複**を別項目で出す
- ✅ 同義の論点は最も情報量の多い1件にまとめて出す

### 絵文字体系（3マーク・1項目1絵文字）

| 絵文字 | カテゴリ | 性質 |
|---|---|---|
| **💰** | アップセル機会＝提案でお金になりそう | チャンス・攻める |
| **🚨** | 競争・離反リスク（取られる／失う危険信号のみ） | 守る・取りこぼし防止 |
| **●** | それ以外の拾うべきタスク・情報（普通のTODO・状況・ニーズ） | 流し見・実務 |

> 早急さ（至急/今週中等）は**絵文字にしない**。`due` に期日として入れる（🏃マークは廃止）。

### 💰（アップセル機会）を付ける基準（最重要・厳密に）

**★ハードゲート：次の3つが【すべて】揃った時だけ💰。1つでも欠けたら ● にする。**
1. **ナイルの財布が新たに開く**（新規発注・予算増額・今やっていない新領域への投資）
2. **顧客が自分の言葉でそれを欲しがっている（顧客pull）** ← 定義は下記で厳格化（2026-07-01）
3. **その機会がナイルの提供領域（SEO/広告/コンテンツ/サイト改善/計測）の中にある**（スコープゲート・2026-07-01）

#### 条件2「顧客pull」の定義 ★2026-07-01 厳格化
顧客pull＝**顧客本人が、自分の口で、発注方向に踏み込む**こと。次のどれかが要る：
- 顧客が自分から〈発注/依頼したい・見積もり/金額・契約・「いつ始める」〉に言及した
- 顧客が新領域・追加投資・新ソリューションを「やってほしい／欲しい」と明示した

**次は顧客pullではない（→●）**：
- ❌ **課題・不満・リソース不足の"共有"だけ**：「困ってる」「不便」「やらなきゃと思ってた」「手が回らない」「知見がない」「できてない」＝困りごとの表明であって発注意欲ではない
- ❌ **コンサル（ナイル）が無償の範囲で助言・運用手順を教えた場面**を有償の商機に膨らませたもの

#### 条件2の前提「話者の峻別」★2026-07-01 追加
💰の根拠にできるのは **顧客（クライアント）本人の発言だけ**。
- ❌ ナイル社員・制作パートナー・代理店側の発言を顧客pullに数えない
- ❌ 「〜もやりますよね？」（相手からの確認質問）、「ナイル側で〜する」（自社の対応説明）を顧客ニーズと誤読しない
- ❌ **エンド顧客がほぼ発言していない議事録**（ナイル×制作パートナーの内部進行・制作すり合わせ会議）では、機会を💰にせず慎重に●へ倒す

#### 条件3「スコープゲート」★2026-07-01 追加
その機会が **ナイルが受注する領域**の中にあるか：
- ❌ **CMS/システム開発・社内システム改修など、ナイルが提供しない領域**の要望は、顧客に強い不満があっても💰にしない（→●）
- ❌ 顧客が「**社内／別ベンダー／制作会社に聞く・相談する**」と言っている＝解決先がナイル外の話は💰にしない

#### amptalkネイティブ「提案機会検知」minute（【ニーズ①】…）の扱い ★2026-07-01
議事録に含まれる amptalk ネイティブの提案機会検知 minute（「〜の余地あり」「商機が高い」等）は **AIの推測であって顧客pullの証拠ではない**。それ自体を💰の根拠にしない。必ず**顧客本人の発注志向の発言**を要約minute/書き起こしで確認する。

**旧来の注意（継続）**：「予算情報あり=機械的に💰」でも「ナイルが提案できそう=💰」でもない。
- ❌ **AIの『提案余地がある／刺さる状況／提案できそう』という推測（AI push）だけでは💰にしない**（→●）
- ❌ **既存契約内のサービス遂行・既存予算内の最適化／配分替え・他社/関連会社が担当の作業は💰にしない**（→●）
- ❌ **「再提案」「提案余地」「予算確保済み」は💰の根拠にしない**

| 💰 付与する（財布が開く × 顧客pull の両方） | 例 |
|---|---|
| ✅ 顧客が新領域への発注意向を明示 | 「Web制作・SNSも1社にまとめて依頼したい、と顧客が要望」 |
| ✅ 予算増額・大きい新規発注を顧客が表明 | 「来期は月XX万規模で新規発注を検討、と顧客が表明」 |
| ✅ 顧客が望む追加提案・新ソリューション・PoC・フェーズ1 | 「内製化サポートを追加で頼みたい、と顧客が希望」 |

| 💰 付けない（→ 正しいマーク） | 例 | マーク |
|---|---|---|
| ❌ **AIの推測だけ（顧客pull無し）** | 「UI/UXでコンサルできる余地あり（顧客は発注要望なし）」 | ● |
| ❌ **既存契約内のサービス遂行** | 「契約内のメルマガ改善アドバイスのヒアリング実施」 | ● |
| ❌ **既存予算内の最適化・配分替え** | 「年間予算確定下でリライト/施策配分を最適化」 | ● |
| ❌ **他社/関連会社が担当の作業** | 「トップのレイアウトは顧客の関連会社が担当」 | ● |
| ❌ **継続契約の維持・死守／2期目継続交渉** | 「現契約を継続するか再提案して判断」 | 🚨 |
| ❌ **確保済み・決裁者承認待ちの既存予算** | 「予算は確保済みだが納得感・承認待ち」 | 🚨 or ● |
| ❌ 金額小さい・高額難・進行管理・事務TODO | 「1本5万・月4本」「LP制作の進捗共有」「請求書送付」 | ● |

### 🚨（競争・離反リスク）を付ける基準（厳密に）

🚨は **「取られる」「失う」の危険信号だけ**。

- **競争リスク（取られる）**: コンペ / RFP / 提案依頼 / 複数社提案 / 競合 / 再提案 / 比較検討 / 外部コンサルティング / 体制見直し / 代理店切替検討
- **離反・成果リスク（失う）**: 流入減 / 順位下落 / 成果不振・KPI未達 / 強い不満・クレーム / 継続・契約の見直し示唆 / 解約懸念 / 危機感

> ⚠️ WBS不足・接点頻度・進行の透明性のような**通常のデリバリー上の不満は🚨にしない**（●にする）。「顧客課題＝全部🚨」は誤り。
> CRO強化・リピート獲得・AI検索対応などの「顧客の関心テーマ」は提案の入口＝💰側に寄せる（成果未達・解約示唆とセットでない限り🚨にしない）。

### ●（普通のタスク・情報）

💰でも🚨でもない、**拾う価値のあるタスク・状況・ニーズ**はすべて●。
ナイル側の「やります」TODO、先方の言質、判断材料になる情報など。

### 優先順位（1項目に複数該当した場合）

**💰 > 🚨 > ●** で1つだけ選ぶ：

- 「コンペで新規領域を提案、早急に対応」 → **💰**（新規領域、最優先）＋ due に早急
- 「コンペRFPの再提案」 → **🚨**（取られるリスク）
- 「成果が出ず継続を見直したいと顧客が示唆」 → **🚨**（失うリスク）
- 「WBS不足・接点頻度に顧客が不満」 → **●**（通常のデリバリー不満。🚨にしない）
- 「至急、提案資料修正」 → **●** ＋ due=「至急」

### 採用基準（拾い漏れ防止優先）

**「拾える情報があれば拾う」がポリシー**：

| 採用する（is_noise=false） | 例 |
|---|---|
| ✅ 数字・金額・ニーズ・時期・状況など何らかの情報あり | 「1本5万、高額難」「Instagram動画週1本希望」 |
| ✅ シグナル語彙がなくても情報あれば拾う（●で） | 「現状の出稿費は月50万」 |

| 除外する（is_noise=true） | 例 |
|---|---|
| ❌ 「未確認」「特になし」「N/A」だけ | 「予算：未確認」 |
| ❌ 完全に空欄・情報なし | 「・」だけ |
| ❌ 進捗確認・社内共有メモのみで判断材料にならない | 「次回MTGで共有予定」 |

▼決定事項・▼タスク<ナイル>・▼BANT は情報あれば原則すべて採用。
▼メモは「マネ＋AMが判断に使える情報」のみ採用（雑談・進捗・社内共有メモのみは is_noise=true）。

### source_section（主な出どころ）

各 distinct 項目に、主な出どころを `decision` / `task_nyle` / `bant` / `memo` のいずれかで付ける
（複数箇所を統合した場合は、最も中心的な出どころを選ぶ）。

### category（領域分類）★v2.2

各項目に **領域カテゴリ** を7枠から**必ず1つ**付ける。通知は領域でグルーピングされる。
施策・成果・話題の**内容**で判断する（出どころセクションではなく中身で分類）。

| category | 該当する内容 |
|---|---|
| **広告** | リスティング / SNS広告 / 運用型広告 / 出稿・入札・広告費 |
| **SEO** | 検索順位 / オーガニック流入 / 内部対策 / 被リンク / コンテンツSEO / AI検索・AIO |
| **サイト改善** | CRO / UI・UX / フォーム改善 / LP / 回遊・直帰・CVR |
| **コンテンツ** | 記事制作 / リライト / 編集 / オウンドメディア運営 |
| **SNS運用** | Instagram・X 等の運用 / 投稿 / アカウント運用（広告ではなくオーガニック運用） |
| **計測・データ基盤** | GA4 / GTM / タグ / ダッシュボード / データ連携・可視化 |
| **その他** | 上記に当てはまらない / 複数領域を横断 / 判断が難しい |

- **広告 vs SNS運用**：費用を払って出稿＝広告。アカウントのオーガニック運用＝SNS運用
- **SEO vs コンテンツ**：検索流入を狙う記事戦略＝SEO。制作・編集・運営の実務＝コンテンツ
- 迷ったら **その他**（無理にこじつけない）

### due（期日）

期日があれば原文の期日表記を `due` に入れる（例 `2026/07/03`、`今週中`、`月末`）。
なければ空文字。早急さもここに入れる（至急/今週中等）。

### label 生成ガイド（全項目に付与）★v2.3で長さ拡張

各項目に **`label`（見出し）** を付ける。**💰/🚨 は親メッセージに `絵文字 label` の1行だけが単独表示**され
（サマリ・期日はスレッド返信に分離）、**● は `● label　【期日】期日` の1行がそのまま親に出る**。
つまり label は「何の話か」の見出しではなく、**単体で内容が伝わる最小限の要約**として機能する必要がある。

- **20〜30字程度**の体言止め（名詞句）。`【】`記号は付けない（Python側が付ける）
- テーマ名だけで終わらせず、**主語（誰の／どの領域の）・対象・狙い**まで一目で分かる具体性を持たせる
- 8〜12字時代の短い見出し（例「AI検索対応の余地」）はNG。「何を・なぜ」まで詰める

| 内容 | ❌ 短すぎる旧label | ✅ 新label（20〜30字） |
|---|---|---|
| EC・コーポレート連携のサイト構造/UX設計を来週中に完了要望 | サイト構造・UX設計の追加提案 | EC×コーポレート連携のサイト構造・UX設計を追加提案 |
| AI検索対策を次フェーズで提案検討、追加提案の余地 | AI検索対応の余地 | 次フェーズでAI検索(AIO)対応コンテンツの追加提案余地 |
| RFP受領後にSEO/コンテンツ施策を再提案（競争局面） | コンペ再提案 | 代理店見直しのRFPでSEO/コンテンツ領域が再提案対象に |
| 成果不振で顧客が継続を見直したいと示唆 | 継続見直しの兆候 | 成果不振を理由に顧客が契約継続の見直しを示唆 |
| 最終提出用ヒアリングシートを社内調整し顧客提出 | ヒアリングシート提出 | 最終ヒアリングシートを社内調整のうえ顧客へ提出 |

### summary 生成ガイド（重要）★v2.2で情報量を厚く

label は「何の話か」を殴る見出し。summary は **「読んだだけで状況が飲み込める」本文**。
label を読んだ人が抱く **「で、何がどうなって・なぜ？」に答える**のが summary の仕事。

**目安 80〜120字・必要なら2文**（簡潔さより情報量優先。ただし冗長・逐語繰り返しはNG）。
次の3点を、**いずれも元データ（要約minute・書き起こし）に実際にある事実の範囲で**織り込む：

1. **① 何が起きたか（状況・経緯）**：誰が・何を・どういう経緯で
2. **② なぜ機会／リスク／タスクなのか（理由・背景）**：💰なら「なぜ攻める価値があるか」、🚨なら「なぜ取られる/失う危険か」。**元データで語られている理由だけ**書く
3. **③ だから何が必要か**：ナイル側に求められるアクションの含意（**元データに根拠がある場合のみ**）

- 元議事録の会議議題・他セクション・amptalk提案機会検知の文脈をフル活用して背景を補う（ただし上記の通り、原文に無い事柄の創作はしない）

## 出力スキーマ（Claude API への期待出力）

**1議事録 = distinct な項目の JSON 配列**（前後にテキスト・コードブロックを付けない）：

```json
[
  {
    "emoji": "💰",
    "label": "20〜30字程度の見出し（体言止め、【】記号なし。単体で内容が伝わる具体性を持たせる）",
    "summary": "情報量厚めのサマリ（80〜120字目安。何がどうなって・なぜ機会/リスクかが分かる。期日表記は含めない）",
    "category": "SEO",
    "source_section": "decision",
    "due": "2026/07/03",
    "is_noise": false
  }
]
```

- `emoji`: `💰` / `🚨` / `●` のいずれか1つ
- `label`: 20〜30字程度の見出し（体言止め、`【】`なし）。💰/🚨は親メッセージにこの1行だけが単独表示されるため、主語・対象・狙いまで分かる具体性を持たせる。全項目に付与
- `summary`: 情報量厚めのサマリ（80〜120字目安・必要なら2文。①何がどうなって②なぜ機会/リスク/タスクか③だから何が必要か、を織り込む。無理な短縮はしない、期日表記は含めない）
- `category`: `広告` / `SEO` / `サイト改善` / `コンテンツ` / `SNS運用` / `計測・データ基盤` / `その他` のいずれか1つ（全項目に必ず付与）
- `source_section`: `decision` / `task_nyle` / `bant` / `memo`
- `due`: 期日表記（あれば。早急さもここ）。なければ空文字
- `is_noise`: true なら出力から除外

## Phase 6: 通知＆スプシ書き込み（Pythonが担当）

### 通知フォーマット（v2.3・親1行圧縮＋スレッド詳細分離／メンションなし版）

1議事録 = **1親メッセージ＋1スレッド返信**。distinct な項目を **領域カテゴリ** でグルーピングして整形する。
親は「一覧性」（何件・何がどのカテゴリにあるか一目で分かる）、スレッドは「詳細を読む用」に役割分担する。

**親メッセージ**
- 冒頭の装飾絵文字（📌・💼）は付けない（ご茶つき防止）。案件名は太字のみ
- カテゴリは固定順（広告 → SEO → サイト改善 → コンテンツ → SNS運用 → 計測・データ基盤 → その他）で表示。**出番ゼロの枠は非表示**
- カテゴリ見出しは太字 `*【○○関連】*`
- カテゴリ内は **💰 → 🚨 → ●** 順
- **💰 機会・🚨 リスク**：`絵文字丸数字 label　【期日】期日` の**1行のみ**（● と同じく期日はここに出す）。サマリ本文はスレッドへ。だからこそ label は20〜30字で単体で内容が伝わる具体性が要る
- **丸数字（①②③…）**は💰/🚨の親↔スレッド対応付け専用。カテゴリ固定順→カテゴリ内優先度順の表示順で連番を振る（Python側が採番）
- **● タスク**：`● label　【期日】期日` の1行（従来通り・変更なし）。**期日付きのみ採用**（期日なし● は通知もスプシも完全除外）
- 期日マーカーは `:date:`（📅＝赤系で🚨と色が被る）を避け、テキストの `【期日】` を使う
- **末尾に太線 `━` を1本**入れて1議事録を締める＝連投時に次MTGとの境目を明確化

```
下記MTGから提案機会を検知しました！

*<案件名>*
*<会議タイトル>*   ← 案件名と異なる時のみ
amptalk URL：<permalink>
────────────────────

*【SEO関連】*
💰① <label>　【期日】<期日表記>
● <label>　【期日】<期日表記>

*【サイト改善関連】*
🚨② <label>

━━━━━━━━━━━━━━━━━━━━   ← 末尾に太線で1議事録を締める
```

**スレッド返信（1レスに💰/🚨のサマリだけまとめる）**
- 親メッセージのts宛に**1レスだけ**投稿（項目ごとに分割しない）
- 対象は **💰/🚨 のみ**（● は親の1行で完結済みのため出さない）
- label・期日は親に出ているためスレッドでは繰り返さない（二重表示防止）。**丸数字＋サマリ本文だけ**
- カテゴリ別に `*【○○関連】*` 見出し→各項目 `絵文字丸数字 サマリ`

```
*【SEO関連】*
💰① <サマリ>

*【サイト改善関連】*
🚨② <サマリ>
```

並べ替え（カテゴリ固定順・カテゴリ内 💰→🚨→●）、丸数字の採番、期日なし●の除外、親/スレッドの振り分けはすべて Python が処理する。

## 実行時の注意

1. 議事録パース・社外/社内振り分け・重複排除（call_id）は Python が担当
2. あなたは **議事録まるごと** を受け取り、**重複を排した distinct な項目リスト** を返す（★過分割を出さない）
3. 絵文字は3マーク（💰/🚨/●）、1項目1絵文字。複数該当時は 💰 > 🚨 > ●。早急さは `due` に入れ🏃マークは作らない
4. 💰は「攻める価値あるアップセル機会」だけ。🚨は「取られる/失う」の危険信号だけ。通常のデリバリー不満は●
5. ●（普通のタスク・情報）も情報があれば拾う（採用基準は緩い）
6. ▼メモは「マネ＋AMが判断に使える情報」のみ採用
7. label（20〜30字程度の体言止め、【】なし・単体で内容が伝わる具体性）を全項目に付与。summary は情報量厚めに（80〜120字目安・何がどうなって/なぜ/だから何、を織り込む）、期日表記は含めない
8. **★v2.2** category（領域・7枠）を全項目に必ず付与。通知は領域でグルーピングされる
9. **★v2.2** 期日のない ● タスクは Python 側で除外される（通知もスプシも出さない）。● を残したい場合は due を入れる
