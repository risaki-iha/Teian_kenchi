# 提案機会検知くん リアルタイム版

議事録転送Bot (`U0B305165M1`) が各案件チャンネルに自動投稿する議事録から、
**提案機会・競争リスク・早急対応** を検知し、通知チャンネルに **議事録単位** で送信する。

## チャンネルID（固定）
- `通知先`: `C0AHUC1VDDK` (`#dxm_提案機会_検知くん`)
- `データソース投稿者`: `U0B305165M1` (議事録転送Bot)

## 目的（思想）
- **MTG の中でアップセル機会・提案機会を拾って通知する**
- 提案すれば取れるはずの案件を取りこぼさない
- マネ＋AMが「ここ提案できる」を把握できる体制をつくる

## 検知対象スコープ
- **対象**: 投稿者が `U0B305165M1`（議事録転送Bot）の投稿のみ

---

## 処理フロー（Pythonが担当する部分）

1. **コール取得（v2＝amptalk文字起こし駆動）**: amptalk API からコールを取得し、`title` プレフィクスで社外だけ抽出（議事録転送Botには依存しない）
2. **パース**: amptalk の「要約minute」（Slack議事録Bot投稿と同形式）を ▼決定事項 / ▼タスク<ナイル> / ▼BANT / ▼メモ にセクション分解。**▼タスク<顧客>は検知対象から除外**
3. **社外/社内振り分け**: コール名（amptalk title）のプレフィクスで判定
4. **期日抽出**: A型（YYYY/MM/DD）/ B型（曖昧表記）
5. **AI判定（このスキルが呼ばれる箇所）**: 各項目に絵文字＋サマリ付与＋ノイズ除外
6. **並べ替え**: セクション内を絵文字優先順位順（🆕 > 🚨 > 🏃 > 装飾なし）に並べ替え（Python側）
7. **通知＆スプシ書き込み**: **🆕/🚨/🏃 が0件の議事録は親投稿もスプシも完全スルー**（emoji0＝機会なし）

## Phase 5: AI判定（あなたの役割）

各議事録項目に対し、**絵文字判定**と**サマリ生成**を行う。

### 絵文字体系（3種類、1項目1絵文字）

| 絵文字 | カテゴリ | 性質 |
|---|---|---|
| **🆕** | アップセル機会・**攻める価値ある**予算系 | チャンス・攻める |
| **🚨** | **競争・離反リスク（取られる／失う危険信号のみ）** | 守る・取りこぼし防止 |
| **🏃** | 早急 | 急ぐ（TODO感あってもOK） |
| （空文字 ""） | 情報はあるが上記に該当しない通常項目（**単なるデリバリー不満もここ**） | 流し読みOK |

> ⚠️ **🚨は厳密に**。「取られる（競合・コンペ・体制見直し）／失う（流入減・順位下落・成果不振・解約示唆・強い離反不満）」の危険信号だけ。
> WBS不足・接点頻度・進行の透明性のような **通常のデリバリー上の不満は🚨にしない**（装飾なしでスレッドへ）。「顧客課題＝全部🚨」は誤り。

### 🆕 を付ける判断基準（重要・厳密に）

「予算情報あり=機械的に🆕」じゃない。**攻める価値ある時だけ🆕**：

| 🆕 付与する | 例 |
|---|---|
| ✅ 大きい予算・新規予算が出る話 | 「月50万〜1000万円規模、高単価案件で再検討」 |
| ✅ 新規領域への拡大シグナル | 「AI検索対策は次フェーズで提案検討」 |
| ✅ 追加提案・新ソリューション・PoC・フェーズ1・来期推進 | 「Instagram動画の内製化サポート希望」 |
| ✅ 内製化・ノウハウ蓄積・再検討意向 | 「内製or外注で運用ノウハウ希望」 |

| 🆕 付けない（装飾なし） | 例 |
|---|---|
| ❌ 予算情報あるが**攻める価値薄い**（金額小さい・高額難） | 「1本5万・月4本（20万）、高額難」 |
| ❌ 既存施策の進行管理 | 「LP制作の進捗共有」 |
| ❌ 事務的TODO | 「請求書送付」「資料整理」 |

### 5カテゴリーとシグナル語彙

| # | カテゴリ | 絵文字 | シグナル語彙の例 |
|---|---|---|---|
| 1 | **アップセル機会** | 🆕 | 追加提案 / 新規領域 / 新ソリューション / フェーズ1 / 来期推進 / 中長期ロードマップ / PoC / 概念実証 / 拡大 / 新施策 / 次フェーズ / 将来検討 / 次のステップ / 内製化 / ノウハウ蓄積 / 内製or外注 / 再検討意向 / **AI検索・AIオーバービュー対応 / CRO強化 / リピート獲得 / 新チャネル**（＝提案の入口） |
| 2 | **競争リスク（取られる）** | 🚨 | コンペ / RFP / 提案依頼 / 複数社提案 / 競合 / 再提案 / 比較検討 / 外部コンサルティング / 体制見直し / 代理店切替検討 |
| 3 | **予算系（攻める価値あり）** | 🆕 | 大きい予算（月XX万以上規模） / 新規予算 / 報酬予算 / 概算費用（金額大） |
| 4 | **離反・成果リスク（失う）** | 🚨 | 流入減 / 順位下落 / 成果不振・KPI未達 / 強い不満・クレーム / 継続・契約の見直し示唆 / 解約懸念 / 危機感 |
| 5 | **早急** | 🏃 | 早急 / 至急 / すぐに / 即時 / 今週中 / 来週中 |

> 🚨はカテゴリ2（取られる）と4（失う）だけ。**CRO強化・リピート獲得・AI検索対応などの「顧客の関心テーマ」は提案の入口＝🆕側**に寄せる（成果未達・解約示唆とセットでない限り🚨にしない）。
> **進行管理・透明性・WBS・接点頻度のような通常のデリバリー上の不満は、🚨でも🆕でもなく装飾なし。**

### 優先順位（複数該当時のロジック）

1項目に複数カテゴリーが該当した場合、**優先順位 🆕 > 🚨 > 🏃** で**1つだけ**選ぶ：

- 「コンペで新規領域を提案、早急に対応」 → **🆕**（新規領域、最優先）
- 「コンペRFPの再提案」 → **🚨**（コンペ・再提案＝取られるリスク）
- 「成果が出ず、継続を見直したいと顧客が示唆」 → **🚨**（離反リスク＝失う）
- 「WBS不足・接点頻度に顧客が不満」 → **空**（通常のデリバリー不満。🚨にしない）
- 「至急、提案資料修正」 → **🏃**（早急のみ）
- 「次回MTGで進捗共有」 → 空（事務的）

### 採用基準（拾い漏れ防止優先・重要）

**「拾える情報があれば拾う」がポリシー**：

| 採用するもの（is_noise=false） | 例 |
|---|---|
| ✅ 数字・金額・ニーズ・時期・状況など**何らかの情報あり** | 「1本5万、高額難」「Instagram動画週1本希望」 |
| ✅ シグナル語彙含まなくても情報あれば拾う（装飾なしで表示） | 「現状の出稿費は月50万」 |
| ✅ 緊急度低くてもニーズ情報あれば | 「SEO強化、AI検索次フェーズ、緊急度低」 |

| 除外するもの（is_noise=true） | 例 |
|---|---|
| ❌ 「未確認」「特になし」「N/A」だけ | 「予算：未確認」「Authority：未確認」 |
| ❌ 完全に空欄・情報なし | 「・」だけ |
| ❌ 進捗確認・社内共有メモのみで判断材料にならない | 「次回MTGで共有予定」 |

▼決定事項・▼タスク（顧客/ナイル）・▼BANT は **情報あれば原則すべて採用**。「特になし」「未確認のみ」だけ除外。

▼メモは「マネ＋AMが議事録元投稿を開かなくても判断に使える情報」のみ採用（雑談・進捗・社内共有メモのみは除外）。

### ラベル生成ガイド（label・新規・全項目に付与）

各項目に **`label`（テーマ見出し）** を付ける。通知では `絵文字【label】サマリ` の形で表示され、
**「何の話か」を最初の数語で殴る**役割。スキャンしやすさの要。

- **8〜12字程度**の体言止め（名詞句）。`【】`記号は Python 側が付けるので **label 本文だけ**返す
- 「何の機会／何のリスク／何のタスクか」を端的に。サマリの要約タイトルと考える
- 装飾なし項目（emoji=""）にも label は付ける

| 項目の内容 | label の例 |
|---|---|
| 事後アンケートに営業項目を追加して架電リスト化を提案できる | 架電リスト化提案 |
| AI検索対策を次フェーズで提案検討、追加提案の余地 | AI検索対応の余地 |
| RFP受領後にSEO/コンテンツ施策を再提案（競争局面） | コンペ再提案 |
| 成果不振で顧客が継続を見直したいと示唆 | 継続見直しの兆候 |
| WBS不足・接点頻度に顧客が不満 | 進行管理への不満 |
| 最終提出用ヒアリングシートを社内調整し顧客提出 | ヒアリングシート提出 |

### サマリ生成ガイド（summary・重要）

**自然な日本語で簡潔に。〜50字目安**（厳格な上限ではない。自然さ最優先）：

- **「何を」「誰が」が分かる**具体的な文章（「いつまでに」は別途【期日】行で出すのでサマリに含めない）
- **抽象動詞（対応・検討・確認）は避け、具体的な目的語を含める**
- 元議事録の **会議タイトル・他セクション本文の文脈**も使って、何の話か明示する
- 体言止めOK、句点なくてもOK
- ⚠️ **だらだら同じことを繰り返さない**。重複・くどい修飾は削る。label と丸かぶりの言い回しも避ける
- ⛔ **無理な短縮・不自然な略語は禁止**（「事後アンケ」「KW」「→」連発など）。文字数のために日本語を壊さない。削るのは"重複"であって"自然さ"ではない
- ⚠️ **サマリには期日表記を含めない**（「2026/06/16まで」「期日：早急」「2026/06/末」等）。Python 側が別途【期日】行で表示するため二重表示になる

#### 良い例 vs 悪い例

| ❌ 悪い | 理由 | ✅ 良い |
|---|---|---|
| 7月着手必須、今週中連絡希望 | 文脈欠如・抽象 | 動画制作プロジェクトの7月着手必須、今週中にマッチ度・スケジュールを連絡希望 |
| 事後アンケに営業項目追加→リスト化し架電 | 不自然な圧縮・略語 | 事後アンケートに営業項目を追加してリスト化し、架電アプローチを追加提案できる |
| 提案機会あり | 中身ゼロ | コンペRFP対応として複数社提案実施、ナイル含む再提案へ |
| AI検索対策を次フェーズで提案検討し、追加提案の余地があり提案できるチャンス | 同義の重複でくどい | AI検索対策を次フェーズで提案検討、追加提案の余地あり |

## Phase 6: 通知＆スプシ書き込み（Pythonが担当）

### 通知フォーマット（メンションなし版）

1議事録 = 1親メッセージ + 1スレッド子メッセージ。

親メッセージは元議事録のセクションを横断し、**絵文字軸の3セクション**（攻める/守る/やる）に再分類する。
各項目は `絵文字【label】サマリ` の1行。**期日がある項目だけ**【期日】行を続ける（期日なしは行ごと省略）。

#### 親メッセージ テンプレート
```
📌 下記MTGから提案機会を検知しました！

💼 *<案件名>*
*<会議タイトル>*   ← 案件名と異なる時のみ
議事録：<permalink>
────────────────────

*▼アップセル機会*
🆕【<label>】<サマリ>
【期日】<期日表記>   ← 期日がある時だけ

*▼リスク警告*
🚨【<label>】<サマリ>

*▼タスク（TODO）*
🏃【<label>】<サマリ>
【期日】<期日表記>
```

装飾なし項目（emoji=""）と ▼メモ採用分は親に載せず、スレッド子メッセージ「▼参考情報」へ。

#### スレッド子メッセージ テンプレート
```
*▼参考情報*

[決定事項]
・【<label>】<サマリ>

[タスク<顧客>]
・【<label>】<サマリ>
```

各セクション内の項目は **絵文字優先順位順（🆕 > 🚨 > 🏃 > 空）** に並び替え済（Pythonが処理）。

### スプシ書き込み

スプシ `1UvbEP78vi10WttBcapWI9_Dpe9nvd7YHv2TLdiemIsA` の「AI検知ログ」シート。

**1議事録項目 = 1スプシ行**。F列「分類タグ」は「絵文字＋セクション名」併記：
- `🆕 決定事項` / `🚨 タスク<ナイル>` / `タスク<顧客>` / `🆕 BANT` / `🚨 メモ`

M列「通知URL」には**親メッセージのpermalink**（同議事録の全行で同じURL）。

## 出力スキーマ（Claude API への期待出力）

```json
[
  {
    "item_id": 0,
    "emoji": "🆕",
    "label": "8〜12字のテーマ見出し（体言止め、【】記号なし）",
    "summary": "自然な日本語の簡潔なサマリ（〜50字目安）",
    "is_noise": false
  }
]
```

- `item_id`: 入力で渡された項目の識別子をそのまま返す
- `emoji`: `🆕` / `🚨` / `🏃` / `""`（空文字）のいずれか1つ
- `label`: 8〜12字程度のテーマ見出し（体言止め、`【】`記号は付けない）。装飾なし項目にも付与
- `summary`: 自然な日本語の簡潔なサマリ（〜50字目安、無理な短縮はしない）
- `is_noise`: true なら出力から除外

期日抽出と社外/社内振り分けは Python側完結なので、AI出力には含めない。

## 実行時の注意

1. 議事録パースは Python（正規表現＋セクション抽出）で確実に
2. AIには「項目本文」に加えて meeting_context を渡す。v2では meeting_context に「**会議タイトル**」と「**amptalk提案機会検知**」（＝文字起こしを amptalk が解析して抽出した【ニーズ】等）が入る。提案機会検知は🆕判定の**有力な根拠**として参照する（ただし🆕の厳密基準＝攻める価値があるかは崩さない／機械抽出を鵜呑みにしない）。サマリも文脈を使って具体化する
3. 絵文字判定は **シグナル語彙の存在チェック→優先順位適用**。1項目1絵文字、複数該当時は 🆕 > 🚨 > 🏃。**🚨は競争リスク（取られる）と離反・成果リスク（失う）だけ。通常のデリバリー不満は🚨にしない**
4. 装飾なし（emoji=""）の項目も**情報があれば通知に含める**（採用基準が緩い、装飾なしでも is_noise=false）
5. ▼メモは「マネ＋AMが判断に使える情報」のみ採用
6. **全項目に label（8〜12字の体言止め、【】なし）を付与**。サマリは自然な日本語で簡潔に（〜50字目安）、文脈と具体性を保ちつつ重複・くどさを削る。無理な短縮・略語は禁止
7. 1回のAPIコールに渡す項目数が多い場合は8項目ずつバッチ処理
