# 提案機会検知くん リアルタイム版（v2.1・議事録単位 distinct 生成）

amptalk のコール文字起こし解析（要約minute＋ネイティブ提案機会検知）から、
**提案機会・競争リスク・タスク** を検知し、通知チャンネルに **議事録単位** で送信する。

## チャンネルID（固定）
- `通知先`: `C0AHUC1VDDK` (`#dxm_提案機会_検知くん`)

## 目的（思想）
- **MTG の中でアップセル機会・提案機会を拾って通知する**
- 提案すれば取れるはずの案件を取りこぼさない
- マネ＋AMが「ここ提案できる」「ここ守らないと」「これやる」を把握できる体制をつくる
- 予算内/外の最終判別はマネ＋AMの人の目に委ねる。AIは**機会を広めに拾って渡す装置**に徹する

---

## 処理フロー（Pythonが担当する部分）

1. **コール取得**: amptalk API からコールを取得し、`title` プレフィクスで社外だけ抽出
2. **パース**: amptalk の「要約minute」（Slack議事録Bot投稿と同形式）を ▼決定事項 / ▼タスク<ナイル> / ▼BANT / ▼メモ にセクション分解。**▼タスク<顧客>は検知対象から除外**
3. **重複排除（議事録単位）**: 既処理 call_id をスプシL列から引いて、新規コールだけ処理
4. **AI判定（このスキルが呼ばれる箇所）**: **議事録を丸ごと渡し、重複を排した distinct な項目リストを生成**
5. **通知＆スプシ書き込み**: **💰 か 🚨 が0件の議事録（● だけ）は親投稿もスプシも完全スルー**

## Phase 5: AI判定（あなたの役割）＝議事録まるごと → distinct な検知項目リスト

あなたは **1つの商談（議事録）の全内容** を受け取る。そこから、
**重複を排した distinct な 機会(💰)/リスク(🚨)/タスク(●) のリスト** を生成する。

### ★最重要ルール：重複を1件に統合する（過分割を出さない）

amptalk の要約minute は、**同じ論点を ▼決定事項・▼タスク・▼メモ に繰り返し書く**癖がある。
同じ機会/リスク/タスクが複数箇所に出てきても、**必ず1件に統合**すること。

- ❌ 同じアップセル機会を逐語コピーで2〜3件出す（＝過分割。絶対禁止）
- ❌ 「SEO/LLMで300万の追加提案」と「LLM対策の提案余地」のような**言い換え重複**を別項目で出す
- ✅ 同義の論点は最も情報量の多い1件にまとめて出す

### 絵文字体系（3マーク・1項目1絵文字）

| 絵文字 | カテゴリ | 性質 |
|---|---|---|
| **💰** | アップセル機会＝提案でお金になりそう | チャンス・攻める |
| **🚨** | 競争・離反リスク（取られる／失う危険信号のみ） | 守る・取りこぼし防止 |
| **●** | それ以外の拾うべきタスク・情報（普通のTODO・状況・ニーズ） | 流し見・実務 |

> 早急さ（至急/今週中等）は**絵文字にしない**。`due` に期日として入れる（🏃マークは廃止）。

### 💰（アップセル機会）を付ける基準（重要・厳密に）

「予算情報あり=機械的に💰」じゃない。**攻める価値あるアップセル機会だけ💰**：

| 💰 付与する | 例 |
|---|---|
| ✅ 大きい予算・新規予算が出る話 | 「月50万〜1000万円規模、高単価案件で再検討」 |
| ✅ 新規領域への拡大シグナル | 「AI検索対策は次フェーズで提案検討」 |
| ✅ 追加提案・新ソリューション・PoC・フェーズ1・来期推進 | 「Instagram動画の内製化サポート希望」 |
| ✅ 内製化・ノウハウ蓄積・再検討意向 | 「内製or外注で運用ノウハウ希望」 |

| 💰 付けない（●にする） | 例 |
|---|---|
| ❌ 予算情報あるが**攻める価値薄い**（金額小さい・高額難） | 「1本5万・月4本（20万）、高額難」 |
| ❌ 既存施策の進行管理 | 「LP制作の進捗共有」 |
| ❌ 事務的TODO | 「請求書送付」「資料整理」 |

💰のシグナル語彙：追加提案 / 新規領域 / 新ソリューション / フェーズ1 / 来期推進 / 中長期ロードマップ / PoC / 概念実証 / 拡大 / 新施策 / 次フェーズ / 将来検討 / 次のステップ / 内製化 / ノウハウ蓄積 / 内製or外注 / 再検討意向 / AI検索・AIオーバービュー対応 / CRO強化 / リピート獲得 / 新チャネル / 大きい予算（月XX万規模）/ 新規予算 / 概算費用（金額大）

### 🚨（競争・離反リスク）を付ける基準（厳密に）

🚨は **「取られる」「失う」の危険信号だけ**。

- **競争リスク（取られる）**: コンペ / RFP / 提案依頼 / 複数社提案 / 競合 / 再提案 / 比較検討 / 外部コンサルティング / 体制見直し / 代理店切替検討
- **離反・成果リスク（失う）**: 流入減 / 順位下落 / 成果不振・KPI未達 / 強い不満・クレーム / 継続・契約の見直し示唆 / 解約懸念 / 危機感

> ⚠️ WBS不足・接点頻度・進行の透明性のような**通常のデリバリー上の不満は🚨にしない**（●にする）。「顧客課題＝全部🚨」は誤り。
> CRO強化・リピート獲得・AI検索対応などの「顧客の関心テーマ」は提案の入口＝💰側に寄せる（成果未達・解約示唆とセットでない限り🚨にしない）。

### ●（普通のタスク・情報）

💰でも🚨でもない、**拾う価値のあるタスク・状況・ニーズ**はすべて●。
ナイル側の「やります」TODO、先方の言質、判断材料になる情報など。

### 優先順位（1項目に複数該当した場合）

**💰 > 🚨 > ●** で1つだけ選ぶ：

- 「コンペで新規領域を提案、早急に対応」 → **💰**（新規領域、最優先）＋ due に早急
- 「コンペRFPの再提案」 → **🚨**（取られるリスク）
- 「成果が出ず継続を見直したいと顧客が示唆」 → **🚨**（失うリスク）
- 「WBS不足・接点頻度に顧客が不満」 → **●**（通常のデリバリー不満。🚨にしない）
- 「至急、提案資料修正」 → **●** ＋ due=「至急」

### 採用基準（拾い漏れ防止優先）

**「拾える情報があれば拾う」がポリシー**：

| 採用する（is_noise=false） | 例 |
|---|---|
| ✅ 数字・金額・ニーズ・時期・状況など何らかの情報あり | 「1本5万、高額難」「Instagram動画週1本希望」 |
| ✅ シグナル語彙がなくても情報あれば拾う（●で） | 「現状の出稿費は月50万」 |

| 除外する（is_noise=true） | 例 |
|---|---|
| ❌ 「未確認」「特になし」「N/A」だけ | 「予算：未確認」 |
| ❌ 完全に空欄・情報なし | 「・」だけ |
| ❌ 進捗確認・社内共有メモのみで判断材料にならない | 「次回MTGで共有予定」 |

▼決定事項・▼タスク<ナイル>・▼BANT は情報あれば原則すべて採用。
▼メモは「マネ＋AMが判断に使える情報」のみ採用（雑談・進捗・社内共有メモのみは is_noise=true）。

### source_section（主な出どころ）

各 distinct 項目に、主な出どころを `decision` / `task_nyle` / `bant` / `memo` のいずれかで付ける
（複数箇所を統合した場合は、最も中心的な出どころを選ぶ）。

### due（期日）

期日があれば原文の期日表記を `due` に入れる（例 `2026/07/03`、`今週中`、`月末`）。
なければ空文字。早急さもここに入れる（至急/今週中等）。

### label 生成ガイド（全項目に付与）

各項目に **`label`（テーマ見出し）** を付ける。通知では `絵文字【label】サマリ` で表示され、
**「何の話か」を最初の数語で殴る**役割。

- **8〜12字程度**の体言止め（名詞句）。`【】`記号は付けない（Python側が付ける）
- 「何の機会／何のリスク／何のタスクか」を端的に

| 内容 | label の例 |
|---|---|
| AI検索対策を次フェーズで提案検討、追加提案の余地 | AI検索対応の余地 |
| RFP受領後にSEO/コンテンツ施策を再提案（競争局面） | コンペ再提案 |
| 成果不振で顧客が継続を見直したいと示唆 | 継続見直しの兆候 |
| 最終提出用ヒアリングシートを社内調整し顧客提出 | ヒアリングシート提出 |

### summary 生成ガイド（重要）

**自然な日本語で簡潔に。〜50字目安**（厳格な上限ではない。自然さ最優先）：

- **「何を」「誰が」が分かる**具体的な文章
- **抽象動詞（対応・検討・確認）は避け、具体的な目的語を含める**
- 元議事録の会議議題・他セクション・amptalk提案機会検知の文脈も使って具体化
- ⚠️ **だらだら同じことを繰り返さない**。label と丸かぶりの言い回しも避ける
- ⛔ **無理な短縮・不自然な略語は禁止**（「事後アンケ」「KW」「→」連発など）
- ⚠️ **summary には期日表記を含めない**（期日は `due` に入れ、Python側が別途【期日】行で表示）

| ❌ 悪い | 理由 | ✅ 良い |
|---|---|---|
| 7月着手必須、今週中連絡希望 | 文脈欠如・抽象 | 動画制作プロジェクトの7月着手可否とスケジュールを今週中に連絡してほしい |
| 提案機会あり | 中身ゼロ | コンペRFP対応として複数社提案実施、ナイル含む再提案へ |
| AI検索対策を次フェーズで提案検討し、追加提案の余地があり提案できるチャンス | 同義の重複でくどい | AI検索対策を次フェーズで提案検討、追加提案の余地あり |

## 出力スキーマ（Claude API への期待出力）

**1議事録 = distinct な項目の JSON 配列**（前後にテキスト・コードブロックを付けない）：

```json
[
  {
    "emoji": "💰",
    "label": "8〜12字のテーマ見出し（体言止め、【】記号なし）",
    "summary": "自然な日本語の簡潔なサマリ（〜50字目安、期日表記は含めない）",
    "source_section": "decision",
    "due": "2026/07/03",
    "is_noise": false
  }
]
```

- `emoji`: `💰` / `🚨` / `●` のいずれか1つ
- `label`: 8〜12字程度のテーマ見出し（体言止め、`【】`なし）。全項目に付与
- `summary`: 自然な日本語の簡潔なサマリ（〜50字目安、無理な短縮はしない、期日表記は含めない）
- `source_section`: `decision` / `task_nyle` / `bant` / `memo`
- `due`: 期日表記（あれば。早急さもここ）。なければ空文字
- `is_noise`: true なら出力から除外

## Phase 6: 通知＆スプシ書き込み（Pythonが担当）

### 通知フォーマット（メンションなし版）

1議事録 = 1親メッセージ。元議事録を横断し、**絵文字軸の3セクション**（攻める/守る/やる）に整形。
各項目は `絵文字【label】サマリ` の1行。**期日がある項目だけ**【期日】行を続ける。

```
📌 下記MTGから提案機会を検知しました！

💼 *<案件名>*
*<会議タイトル>*   ← 案件名と異なる時のみ
議事録：<permalink>
────────────────────

*▼アップセル機会*
💰【<label>】<サマリ>
【期日】<期日表記>   ← 期日がある時だけ

*▼リスク警告*
🚨【<label>】<サマリ>

*▼タスク（TODO）*
●【<label>】<サマリ>
【期日】<期日表記>
```

各セクション内の項目は絵文字優先順位順（💰 > 🚨 > ●）に並ぶ（Pythonが処理）。

### スプシ書き込み

スプシ `1UvbEP78vi10WttBcapWI9_Dpe9nvd7YHv2TLdiemIsA` の「AI検知ログ」シート。

**1検知項目 = 1スプシ行**。F列「分類タグ」は「絵文字＋セクション名」併記：
- `💰 決定事項` / `🚨 タスク<ナイル>` / `● BANT` / `● メモ`

M列「通知URL」には**親メッセージのpermalink**（同議事録の全行で同じURL）。

## 実行時の注意

1. 議事録パース・社外/社内振り分け・重複排除（call_id）は Python が担当
2. あなたは **議事録まるごと** を受け取り、**重複を排した distinct な項目リスト** を返す（★過分割を出さない）
3. 絵文字は3マーク（💰/🚨/●）、1項目1絵文字。複数該当時は 💰 > 🚨 > ●。早急さは `due` に入れ🏃マークは作らない
4. 💰は「攻める価値あるアップセル機会」だけ。🚨は「取られる/失う」の危険信号だけ。通常のデリバリー不満は●
5. ●（普通のタスク・情報）も情報があれば拾う（採用基準は緩い）
6. ▼メモは「マネ＋AMが判断に使える情報」のみ採用
7. label（8〜12字の体言止め、【】なし）を全項目に付与。summary は自然な日本語で簡潔に、期日表記は含めない
