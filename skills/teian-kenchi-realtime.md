# 提案機会検知くん リアルタイム版

議事録転送Bot (`U0B305165M1`) が各案件チャンネルに自動投稿する議事録から、
**アップセル機会** および **受注済み案件のナイル側TODO** を検知し、通知チャンネルに送信する。

## チャンネルID（固定）
- `通知先`: `C0AHUC1VDDK` (`#dxm_提案機会_検知くん`)
- `データソース投稿者`: `U0B305165M1` (議事録転送Bot / minutes-forward-bot)

## 検知対象スコープ
- **対象**: 投稿者が `U0B305165M1`（議事録転送Bot）の投稿のみ
- **対象外**: それ以外の投稿（手動投稿・他Bot投稿はすべて対象外）

## 設計の重心
- **主役は 🟢🟡 アップセル機会検知**。マネ＋AMが取りこぼさないようにする
- 🔵 ナイル側TODOは副次的価値（記録目的、リマインダーは別フェーズ）
- 凡庸な「全TODOを通知」ではなく、**重要シグナルをタグで色分け**して伝える

## 処理フロー

### Phase 1: 議事録投稿スキャン（Python: `scan_minutes_bot_posts`）

`mcp__claude_ai_Slack__slack_search_public_and_private` で `from:<@U0B305165M1>` の投稿を取得。前回スキャン以降の新規投稿のみが対象。

### Phase 2: パース（Python: `parse_minutes`）

投稿テキストから「要約:」以降「関係構築・心理的距離:」直前までを抽出。以下の構造でセクション分解する：

- `meeting_title`: `<会議議題>` の本文
- `call_name`: 投稿1行目「コール名:」の値
- `decisions`: ▼決定事項 の各行
- `tasks_customer`: ▼タスク `<顧客>` の各行
- `tasks_nyle`: ▼タスク `<ナイル>` の各行
- `memo`: ▼メモ の各行

### Phase 3: 社外/社内振り分け（Python: `classify_meeting_type`）

`call_name` に基づき3段階で判定：

1. **明示プレフィクス**: `【社外】` `社外_` `【外部】` → external / `【社内】` `社内_` `【内部】` → internal
2. **【確定】プレフィクス**: スケジュール調整済み確定MTG = 基本 external
3. **プレフィクスなし**: AI判定（議題＋▼メモ＋▼タスク<顧客> から推定）

### Phase 4: 期日抽出（Python: `extract_due_date`）

各項目の文末括弧から期日を抽出。

- **A型（確定日が取れる）**：`（2026/06/05〆）` `（2026/06/17まで）` `（期日：2026/06/05）` `（2026/06/03までに）` 等
  → `YYYY-MM-DD` で正規化。複数日付があれば最も遅い日付。
- **B型（確定日が取れない）**：曖昧 (`末/上旬/中旬/下旬`) / 月単位 / 相対 (`受領後1週間以内`) / イベント駆動 (`随時/タスク完了時`) / 状態 (`進行中`) / 緊急 (`早急`) / 期日未定
  → 原文をそのまま `due_raw` に保存。リマインダー対象外（リマインダー機能は別フェーズ）。

### Phase 5: AI判定（このスキルが呼ばれる箇所）

各議事録項目（▼決定事項 / ▼タスク<顧客> / ▼タスク<ナイル> / ▼メモ）に対し、**分類タグ判定**と**サマリ生成**を行う。

#### 分類タグの定義（4タグ＋早急の追加付与）

| タグ | 対象 | シグナル語彙の例 |
|---|---|---|
| 🟢 **アップセル機会**（ポジ） | 新規領域への拡大、追加提案、将来予算化の兆し | 追加提案 / 新規領域 / 新ソリューション / フェーズ1 / 来期推進 / 中長期ロードマップ / PoC / 概念実証 / 拡大 / 新たな施策 |
| 🟡 **競争・リスク警告**（半ポジ半ネガ） | コンペ・他社提案・既存顧客の動揺 | コンペ / RFP / 提案依頼 / 複数社提案 / 競合 / 再提案 / 比較検討 / 外部コンサルティング / 体制見直し |
| 🔵 **ナイル側TODO**（中立・実務） | ▼タスク<ナイル> の項目（▼タスク<ナイル>セクションに記載されたものは原則これ） | セクション位置で判定 |
| ⚪ **顧客アクション参考**（中立・情報） | ▼タスク<顧客> の項目 | セクション位置で判定 |
| 🔥 **早急**（追加付与） | 即対応必要 | 早急 / 至急 / すぐに / 即時 |

#### タグ判定のルール

1. **▼タスク<ナイル>** の全項目は必ず 🔵 を付与
2. **▼タスク<顧客>** の全項目は必ず ⚪ を付与
3. **▼決定事項** / **▼メモ** から **シグナル語彙** を検出した項目に 🟢 または 🟡 を付与
4. 🟢 と 🟡 は**排他ではない**。「コンペで新規領域を提案」なら 🟢🟡 両方付与してよい
5. 🔵 と 🟢/🟡 の併存もOK。▼タスク<ナイル> の中に「コンペ用RFP対応」のように競争シグナルがあれば 🟡🔵 のように付与
6. 🔥 は他タグに追加で付ける（例：🟡🔥）
7. **シグナル語彙に該当しない情報共有のみ**の▼メモ項目は出力配列から除外（ノイズ扱い）

#### サマリ生成ガイド

- **40字程度の1行**で「何が起きたか」「マネ＋AMが取るべきアクション」が一目で分かる文を生成
- 抽象的な「対応が必要」「確認したい」ではなく**具体的に**
- 体言止めOK、句点なくてもOK

良い例：
- `競合含むコンペ形式RFP提案へ。複数社で次回提案実施`
- `テンプレからDTI誘導バナー新設を検討推進、新規領域シグナル`
- `RFP受領後、SEO/コンテンツ施策の再提案（社外要請）`

悪い例（凡庸）：
- `提案機会あり` ← 何の提案か分からない
- `ナイル側で対応が必要` ← 何を対応するか分からない

#### ノイズ除外

以下はAI出力配列に含めない（=スプシ・通知に出さない）：
- ▼メモのうち「進捗確認・雑談・社内共有メモ」のみで提案機会に関係しないもの
- 既に完結済みの事項（「実施済み」「完了報告」など）
- ▼決定事項のうち「来週の会議時間調整」など事務的なもの
- 「特になし」「特段なし」など内容のない記載

⚠️ **▼タスク<ナイル> と ▼タスク<顧客> の項目は原則すべて出力する**（実務TODOは記録目的でも残す）。除外するのは「特になし」など空の記載のみ。

## 出力スキーマ（Claude API への期待出力）

```json
[
  {
    "item_id": "string",
    "tags": ["🟡", "🔵"],
    "summary": "40字程度の1行サマリ",
    "is_noise": false
  }
]
```

- `item_id`: 入力で渡された項目の識別子（Pythonが付与）
- `tags`: 1つ以上の分類タグの配列（🟢/🟡/🔵/⚪/🔥）
- `summary`: 40字程度の1行
- `is_noise`: true なら出力配列から除外（=スプシ・通知に出さない）

期日抽出（due_date / due_raw）と社外/社内振り分け（meeting_type）は **Python側で完結する**ためAI出力には含めない。

## 実行時の注意

1. 議事録パースは Python（正規表現＋セクション抽出）で確実に。AIにはパース済み構造を渡す
2. 分類タグ判定は **シグナル語彙の存在チェックを優先**。語彙なしの▼決定事項/▼メモは 🟢🟡 を付けない（▼タスク以外は記録不要）
3. **▼タスク<ナイル> は必ず 🔵 を付与する**（例外なし、▼タスク<ナイル>セクションに書かれている時点で実務TODOとして扱う）
4. サマリは必ず項目本文の情報を抽象化せずに具体的に書く。マネ＋AMが本文を読まなくても判断できるレベル
5. 1回のAPIコールに渡す項目数が多い場合は8項目ずつバッチ処理（既存 detector.py の `EVALUATE_BATCH_SIZE` パターン踏襲）
