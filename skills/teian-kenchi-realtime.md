# 提案機会検知くん リアルタイム版

議事録転送Bot (`U0B305165M1`) が各案件チャンネルに自動投稿する議事録から、
**提案機会・競争リスク・早急対応** を検知し、通知チャンネルに **議事録単位** で送信する。

## チャンネルID（固定）
- `通知先`: `C0AHUC1VDDK` (`#dxm_提案機会_検知くん`)
- `データソース投稿者`: `U0B305165M1` (議事録転送Bot / minutes-forward-bot)

## 検知対象スコープ
- **対象**: 投稿者が `U0B305165M1`（議事録転送Bot）の投稿のみ
- **対象外**: それ以外の投稿（手動投稿・他Bot投稿はすべて対象外）

## 設計の重心
- **主役は提案機会の検知**。マネ＋AMが「攻める🆕 / 守る🚨 / 急ぐ🏃」を一目で判別できる体系
- 凡庸な「全TODOを通知」ではなく、**重要シグナル含む項目だけを絵文字で目立たせる**
- 議事録単位で 1親メッセージ + 1スレッド子メッセージ にまとめてマネ＋AMの読み込み負荷を最小化

## 処理フロー

### Phase 1: 議事録投稿スキャン（Python: `scan_minutes_bot_posts`）

`from:<@U0B305165M1>` の投稿を Slack 検索で取得。

### Phase 2: パース（Python: `parse_minutes`）

投稿テキストから「要約:」以降「関係構築・心理的距離:」直前までを抽出。以下の構造でセクション分解：

- `meeting_title`: `<会議議題>` の本文
- `call_name`: 投稿1行目「コール名:」の値
- `decisions`: ▼決定事項 の各行
- `tasks_customer`: ▼タスク `<顧客>` の各行
- `tasks_nyle`: ▼タスク `<ナイル>` の各行
- `memo`: ▼メモ の各行

### Phase 3: 社外/社内振り分け（Python: `classify_meeting_type`）

`call_name` に基づき3段階で判定：

1. **明示プレフィクス**: `【社外】` `社外_` `【外部】` → external / `【社内】` `社内_` `【内部】` → internal
2. **【確定】プレフィクス**: スケジュール確定MTG = 基本 external
3. **プレフィクスなし**: AI判定（議題＋▼メモ＋▼タスク<顧客> から推定）

### Phase 4: 期日抽出（Python: `extract_due_date`）

- **A型（確定日が取れる）**：`（2026/06/05〆）` `（2026/06/17まで）` `（期日：2026/06/05）` 等
  → `YYYY-MM-DD` で正規化
- **B型（確定日が取れない）**：曖昧 (`末/上旬/中旬/下旬`) / 月単位 / 相対 / イベント駆動 / 状態 / 緊急 / 期日未定
  → 原文をそのまま `due_raw` に保存

### Phase 5: AI判定（このスキルが呼ばれる箇所）

各議事録項目に対し、**絵文字判定**と**サマリ生成**を行う。

#### 絵文字体系（3種類、1項目1絵文字）

| 絵文字 | カテゴリ | 含む語彙 |
|---|---|---|
| **🆕** | アップセル機会 ＋ 予算系 | 追加提案 / 新規領域 / 新ソリューション / フェーズ1 / 来期推進 / 中長期ロードマップ / PoC / 概念実証 / 拡大 / 新施策 / 予算 / 月額XX万円 / 報酬予算 / 概算費用 |
| **🚨** | 競争・リスク警告 ＋ 顧客課題 | コンペ / RFP / 提案依頼 / 複数社提案 / 競合 / 再提案 / 比較検討 / 外部コンサルティング / 体制見直し / 危機感 / 流入減 / 順位変動 / CRO強化 / リピート獲得 / AIオーバービュー / 検索行動変化 |
| **🏃** | 早急 | 早急 / 至急 / すぐに / 即時 / 早期 |
| （空文字） | 通常項目 | 既存施策の進行管理・事務作業・進捗確認・スケジュール調整 |

#### 優先順位（複数該当時のロジック）

1項目に複数カテゴリーが該当した場合、**優先順位 🆕 > 🚨 > 🏃** で**1つだけ**選ぶ。

例：
- 「コンペで新規領域を提案、早急に対応」 → **🆕**（新規領域、最優先）
- 「コンペRFPの再提案」 → **🚨**（コンペ・再提案）
- 「至急、提案資料修正」 → **🏃**（早急のみ）
- 「次回MTGで進捗共有」 → 空（事務的）

#### サマリ生成ガイド

- **40字程度の1行**で「何が起きたか」「マネ＋AMが取るべきアクション」が一目で分かる文を生成
- 抽象的な「対応が必要」ではなく**具体的に**
- 体言止めOK、句点なくてもOK

良い例：
- `競合含むコンペ形式RFP提案へ。複数社で次回提案実施で合意`
- `テンプレからDTI誘導バナー新設を検討推進、新規領域シグナル`
- `RFP受領後、SEO/コンテンツ施策の再提案`

悪い例（凡庸）：
- `提案機会あり` ← 何の提案か分からない
- `ナイル側で対応が必要` ← 何を対応するか分からない

#### ▼メモのフィルタリング（重要）

▼メモは「マネ＋AMが議事録元投稿を開かなくても判断に使える情報」のみ採用する。

**採用する例：**
- 顧客課題・危機感の表明
- コンペや競合の事情
- 予算情報・金額感
- 施策方向性の議論
- 提案の入口になりそうな顧客発言

**除外する例（is_noise=true）：**
- 進捗確認・社内共有メモ
- 雑談・スケジュール調整
- 完結済み事項（「実施済み」「完了報告」）
- 「特になし」など空の記載

#### ▼決定事項・▼タスクのフィルタリング

▼決定事項・▼タスク は原則すべて出力する（is_noise=false）。「特になし」「特段なし」のみ除外。

## 出力スキーマ（Claude API への期待出力）

```json
[
  {
    "item_id": 0,
    "emoji": "🆕",
    "summary": "40字程度の1行サマリ",
    "is_noise": false
  }
]
```

- `item_id`: 入力で渡された項目の識別子（Pythonが付与）
- `emoji`: `🆕` / `🚨` / `🏃` / `""`（空文字、装飾なし）のいずれか1つ
- `summary`: 40字程度の1行
- `is_noise`: true なら出力から除外（=スプシ・通知に出さない）

期日抽出（due_date / due_raw）と社外/社内振り分け（meeting_type）は **Python側で完結する** ためAI出力には含めない。

## 実行時の注意

1. 議事録パースは Python（正規表現＋セクション抽出）で確実に。AIにはパース済み構造を渡す
2. 絵文字判定は **シグナル語彙の存在チェック→優先順位適用**。1項目1絵文字（複数該当時は 🆕 > 🚨 > 🏃 の順）
3. 装飾なし（emoji=""）の項目も**▼決定事項・▼タスクなら通知に含める**。装飾なしでも is_noise=false が原則
4. ▼メモは「マネ＋AMが判断に使える情報」だけ採用、それ以外は is_noise=true で除外
5. サマリは具体的に。マネ＋AMが本文を読まなくても判断できるレベルで
6. 1回のAPIコールに渡す項目数が多い場合は8項目ずつバッチ処理
